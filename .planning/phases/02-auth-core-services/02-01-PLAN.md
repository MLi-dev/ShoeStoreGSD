  <behavior>
    register(email, password) -> dict:
      - Reject duplicate email with code "EMAIL_ALREADY_EXISTS", retryable=False
      - Hash password with passlib CryptContext(schemes=["bcrypt"])
      - Create User(id=str(uuid4()), email=email, password_hash=hash, created_at=datetime.utcnow().isoformat())
      - Store in users_db keyed by user.id
      - Return {"success": True, "data": {"user_id": user.id, "email": user.email}}

    login(email, password) -> dict:
      - Find user by iterating users_db.values() to match email (no index needed — demo)
      - User not found: return {"success": False, "code": "INVALID_CREDENTIALS", "message": "Invalid email or password", "retryable": False}
      - Wrong password: same INVALID_CREDENTIALS response (never distinguish user-not-found from wrong-password)
      - On success: encode JWT with payload {"user_id": user.id, "exp": utcnow + timedelta(minutes=JWT_EXPIRE_MINUTES)}
      - Return {"success": True, "data": {"token": jwt_str, "user_id": user.id}}

    verify_token(token) -> dict:
      - Decode with jwt.decode(token, JWT_SECRET, algorithms=[JWT_ALGORITHM])
      - On jwt.ExpiredSignatureError: return {"success": False, "code": "TOKEN_EXPIRED", "message": "Token has expired", "retryable": False}
      - On jwt.InvalidTokenError: return {"success": False, "code": "INVALID_TOKEN", "message": "Token is invalid", "retryable": False}
      - On success: return {"success": True, "data": {"user_id": payload["user_id"]}}

    reset_request(email) -> dict:
      - Always return success (do not reveal whether email exists — demo note: token IS returned in body)
      - If email exists in users_db: generate token = str(uuid4()), store reset_tokens_db[email] = {"token": token, "expires_at": utcnow + timedelta(minutes=15)}
      - Return {"success": True, "data": {"token": token if email found else None, "message": "If the email exists, a reset token has been issued"}}
      - NOTE: Returning the token directly in the response is intentional for demo purposes (no email server). Add a comment in the code.

    reset_confirm(token, new_password) -> dict:
      - Iterate reset_tokens_db to find entry where entry["token"] == token
      - Not found: return {"success": False, "code": "INVALID_RESET_TOKEN", "message": "Reset token not found or expired", "retryable": False}
      - Found but datetime.utcnow() > entry["expires_at"]: delete entry, return same INVALID_RESET_TOKEN error
      - On success: hash new_password, update user.password_hash in users_db (find user by email key), delete reset_tokens_db entry
      - Return {"success": True, "data": {"message": "Password updated successfully"}}
  </behavior>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → register() | Untrusted email/password strings from future route handlers |
| caller → login() | Untrusted credentials; must not reveal user existence |
| caller → verify_token() | Untrusted JWT string; must handle tampering and expiry |
| caller → reset_confirm() | Untrusted token string; must enforce expiry |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | login() | mitigate | INVALID_CREDENTIALS returned for both wrong-password and no-user — prevents user enumeration |
| T-02-02 | Tampering | verify_token() | mitigate | jwt.decode() verifies HS256 signature; InvalidTokenError caught and returned as structured failure |
| T-02-03 | Repudiation | reset_request() | accept | Demo app; no audit log required at this layer |
| T-02-04 | Information Disclosure | register() | mitigate | Only user_id and email returned on success; password_hash never appears in any return value |
| T-02-05 | Information Disclosure | reset_request() | mitigate | Response body is identical whether email exists or not (token is None when no user found) |
| T-02-06 | Elevation of Privilege | reset_confirm() | mitigate | Expired tokens deleted on detection; token only valid 15 min per D-04 |
</threat_model>
